## [4] STREAMLIT SECURITY HARDENING
- **Secret Management (st.secrets)**: NEVER save passwords or API keys in plaintext. Use `.streamlit/secrets.toml` locally and ENSURE it is tracked in `.gitignore`. For production, use environment variables or native Cloud Secrets panels.
- **Robust Authentication**: Implement `streamlit-authenticator` utilizing bcrypt for password hashing and secure cookies for session persistence, or utilize an upstream SSO.
- **Network & Headers (Reverse Proxy)**: Port 8501 is plaintext HTTP. Do not expose it to the internet directly. Deploy Streamlit behind a Reverse Proxy (Nginx/Apache) for SSL/TLS termination (HTTPS) and enforce security headers: Content Security Policy (CSP), X-Frame-Options, and X-Content-Type-Options.
- **Input Validation (Injection)**: Sanitization is mandatory. Keep rigorous control over `st.text_input` and `st.file_uploader`. NEVER pass unfiltered inputs to `os.system()` or `subprocess` to prevent Command Injection.
- **HTML & XSS (Security Paradox)**: `st.write` escapes HTML by default. Do NOT use `unsafe_allow_html=True` unless strictly necessary and perfectly sanitized to avoid Cross-Site Scripting (XSS).
- **Sandboxing**: Protect the host environment. Run the application in a Docker container utilizing a non-root user and restrict physical resources constraints.
- **Global Config (`config.toml`) Tuning**:
  - `browser.gatherUsageStats = false` (Privacy: block Streamlit telemetry).
  - `server.enableCORS = false` (Block cross-domain requests unless strictly required).
  - `server.enableXsrfProtection = true` (Defense against Cross-Site Request Forgery).
  - `server.maxUploadSize = 200` (or less) to prevent memory RAM exhaustion (DoS attacks) via `st.file_uploader`.
